原因剖析

于是我就猜想，这个崩溃肯定与Activity启动过程中某个类实例化的时间有关。

从这段异常信息来看，异常类是ViewRootImpl$CalledFromWrongThreadException，并且可以根据第三行的提示可以看出，这个异常是在ViewRootImpl类的checkThread()方法里面抛出的，那接下来我们来看看在这个方法里面都做了些什么工作：

这个方法主要是检查一下当前线程(Thread.currentThread())是不是"创建视图层次结构的原始线程(mThread)"，所谓创建视图层次结构的原始线程，就是指创建ViewRootImpl的线程，如果创建ViewRootImpl的线程和当前线程不同，则会抛出异常:CalledFromWrongThreadException。

因为一般默认ViewRootImpl都是由系统自动创建的，所以mThread一般来说都是主线程，ViewRootImpl 中和 UI 操作相关的方法都调用了 checkThread() 来检测当前线程是主线程。这就是为什么不能在子线程中操作 UI 的原因。

但是话又说回来，为什么第一次测试时，可以成功在子线程修改TextView的文本呢？那就只有一种解释了：执行setText()方法时，ViewRootImpl还未被创建，因此checkThread()方法也就无从调用了，所以也就不会抛出异常了，我们接下来去看一下ViewRootImpl是什么时候被创建的。
