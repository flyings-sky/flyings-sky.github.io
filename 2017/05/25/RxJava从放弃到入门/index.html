正文

RxJava到底是什么？

RxJava是一个基于观察者模式思想，基于响应事件驱动的一个异步执行库。

为什么要使用RxJava?

RxJava最明显的特点是:简洁和链式调用，避免在进行网络请求时发生多层嵌套的情况。

RxJava可以使用各种操作符对事件序列进行变换，所谓变换，就是将事件序列中的对象或整个序列进行加工处理，转换成不同的事件或事件序列。

RxJava还可以实现线程的自由控制以及随意切换。

RxJava中的四个概念

Observable(被观察者)、Observer(观察者)、subscribe(订阅)、事件。

Observable和Observer通过subscribe()方法实现订阅关系，Observable通过发出事件来通知Observer。

RxJava中的回调方法

onNext():用于响应从Observable发送来的事件。

onCompleted():事件队列中的事件全部处理完毕的触发。

onError():事件处理过程中出现异常时触发，同时事件队列自动终止，事件队列中的事件全部舍弃。

在一个正确运行的事件序列中，onCompleted()和onError()有且只有一个会被调用，而且都是作为事件序列中的最后一个事件被调用。

### RxJava的基本实现

创建观察者Observer

除了Observer接口之外，RxJava还内置了一个实现了Observer的抽象类:Subscriber，在Subscribe订阅的过程中，Observer会先在内部被转换成一个Subscriber再使用。Subscriber对Observer接口进行了一些扩展，添加了onStart()和unsubscribe()方法。

onStart():会在subscribe刚开始事件还未发送之前被调用，用于做一些准备工作，如果对准备工作的线程有要求，这个方法就不适用了，因为这个方法是在subscribe发生的线程被调用，不能单独指定线程。可以使用doOnSubscribe()方法来在指定的线程来做准备工作。

unsubscribe():这个方法用于取消订阅。一般在这个方法前，会使用isUnsubscribed()先判断一下当前状态。这个方法非常重要，因为在subscribe()之后，Observable会持有Subscriber的引用，这个引用如果不能及时被释放，将有内存泄漏的风险。

除此之外，使用下面的方式也可以创建Observable

  • just(T…):将传入的参数依次发送出来
  • from(T[])/from(Iterable<? extends T>):将传入的数组或Iterable拆分成具体对象后，依次发送出来

RxJava的线程控制

在不指定线程的情况下，RxJava遵循的是线程不变原则，即:在哪个线程调用subscribe()，就在哪个线程生产事件，就在哪个线程消费事件。如果需要切换线程，就需要用到Scheduler(调度器)。

  • Schedulers.immediate(): 直接在当前线程运行，相当于不指定线程。这是默认的 Scheduler。
  • Schedulers.newThread(): 总是启用新线程，并在新线程执行操作。
  • Schedulers.io(): I/O 操作（读写文件、读写数据库、网络信息交互等）所使用的 Scheduler。行为模式和 newThread() 差不多，区别在于 io() 的内部实现是是用一个无数量上限的线程池，可以重用空闲的线程，因此多数情况下 io() 比 newThread() 更有效率。不要把计算工作放在 io() 中，可以避免创建不必要的线程。
  • Schedulers.computation(): 计算所使用的 Scheduler。这个计算指的是 CPU 密集型计算，即不会被 I/O 等操作限制性能的操作，例如图形的计算。这个 Scheduler 使用的固定的线程池，大小为 CPU 核数。不要把 I/O 操作放在 computation() 中，否则 I/O 操作的等待时间会浪费 CPU。
  • 另外， Android 还有一个专用的 AndroidSchedulers.mainThread()(这个方法是存在于RxAndroid包中的)，它指定的操作将在 Android 主线程运行。

RxJava中的线程控制主要是两个方法:subscribeOn()和observeOn()

subscribeOn():指定subscribe()所发生的线程，即 Observable.OnSubscribe被激活时所处的线程；

observeOn():指定的是当前执行的Observable所对应的直接下级Subscriber所 运行的线程，叫做事件消费进程；

线程的多次切换:

虽然subscribeOn()对事件处理的流程没有影响，但在流程之前却是可以利用的。在前面写过，onStart方法的执行线程是subscribeOn()指定的线程的，而subscribeOn()线程指定的一般都是进行网络请求的子线程，所以如果想要在onStart()方法中，进行UI操作时，会出现问题，还有另外一个方法Observable.doOnSubscribe()，这个方法也是在subscribe()调用后，事件发送前执行的，它与onStart方法的区别是，可以给它指定执行的线程，默认情况下，doOnSubscribe()执行在subscribe()发生的线程，但是如果在doOnSubscribe()之后有subscribeOn()的话，它将执行在离他最近的subscribeOn()所指定的线程。
