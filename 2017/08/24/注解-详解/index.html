什么是注解？

注解(也被称为元数据)为我们在代码中添加信息提供了一种形式化的方法，使我们可以在稍后某个时刻非常方便地使用这些数据。

注解是注释的升级版，它用于向编译器、虚拟机等解释说明一些事情。通过注解，程序开发人员可以在不改变原有逻辑的情况下，在源代码文件中嵌入一些补充信息。代码分析工具可以通过解析这些注解获取到这些补充信息。

优点

  • 注解使得我们能够以将由编译器来测试和验证的格式，存储有关程序的额外信息。
  • 有助于减轻编写模板代码的负担(如ButterKnife框架)。
  • 更加干净易读的代码以及可以在编译期进行类型检查。

缺点

在大量使用基于反射的运行时注解时，执行效率会比较低。

注解的类型

标记注解:没有元素的注解

  • @Override:限定重写父类的方法，它可以强制一个子类必须覆盖父类的方法，写在子类的方法上，在编译器，编译器检查这个方法，保证父类包含该父类重写方法，否则编译出错。该注解只能修饰方法，在编译期被读取。
  • @Deprecated:表示已过时，编译时读取，编译器编译到过时的元素会给出警告。
  • @SuppressWarnings:抑制编译警告
  • @SafeVarargs(java 7新增):去除"堆污染(把一个不带泛型的对象赋给一个带泛型的变量就会发生堆污染)"警告
  • @FunctionInterface(java 8新增):修饰函数式接口(接口中只有一个抽象方法(可以包含多个默认方法或static方法))
  • @Repeatable - Java 8 开始支持，标识某注解可以在同一个声明上使用多次。

元注解:

  • @Retention用于指定被修饰的Annotation可以保留多长时间，可以指定的值：
    • RetentionPolicy.CLASS:注解在class文件中可用，在类加载的时候丢弃。默认为该类型。
    • RetentionPolicy.SOURCE:注解只保留在源代码中(.java文件中)，在编译阶段丢弃。
    • RetentionPolicy.RUNTIME:编译器在运行期也将保留注解，因此这种注解方式可以通过反射机制读取注解的信息。
  • @Target用于指定注解可以用于修饰哪些程序单元(值由ElementType来指定)。
  • @Documented用于指定被修饰的Annotation将被javadoc工具提取成文档。
  • @Inherited允许子类继承父类中的注解。

自定义注解

使用@interface关键字

可以给注解添加成员变量，并且使用default来设置初始值。

提取注解Annotation的信息

使用自定义的Annotation修饰了类、方法、成员变量等程序元素之后，这些Annotation不会自己生效，必须由开发者通过API来提取并处理Annotation信息。

注解处理器

当注解处理器生成一个新的源文件时，该文件会在新一轮的注解处理中接受检查。该工具会一轮一轮的处理，直到不再有新的源文件产生为止。然后他再编译所有的源文件。

使用apt处理编译时注解(jdk1.7之前)

apt是用来操作Java源文件的，默认情况下，apt会在处理完源文件后编译它们。每一个自定义注解都需要一个注解处理器，使用apt工具可以很容易的将多个注解处理器组合在一起，此外还可以添加监听器，并在一轮注解处理过程结束的时候收到通知信息。

在使用apt时，想要编译这一节的注解处理器，必须要将tools.jar设置在你的classpath中，这个工具类库同时还包含了com.sun.mirror.*接口。

通过使用AnnotationProcessorFactory,apt能够为每一个它发现的注解生成一个正确的注解处理器。当你使用apt时，必须指明一个工厂类，或者指明能够找到apt所需的工厂类的路径。在使用apt生成注解处理器时，我们无法利用java的反射机制，因为我们操作的是源代码，而不是编译后的类。使用mirror API能够解决这个问题，它使我们能够在未经编译的源代码中查看方法，域以及自定义类型。

使用apt处理编译时注解的步骤：

  • 自定义一个编译时注解
  • 在一个类中使用这个编译时注解
  • 创建一个自定义的注解处理器实现AnnotationProcessor接口(的process()方法)，该处理器的构造方法以AnnotationProcessorEnvironment对象为参数。通过该对象，我们就能知道apt正在处理的所有类型，并且可以通过该对象获取到Messager对象和Filer对象那个。Messager对象可以用来向用户报告信息(处理过程中的发生的错误，及错误在源代码中的行数等)。Filer是一种PrintWriter，我们可以通过它创建新的文件。不使用普通的PrintWriter而是使用Filer对象的主要原因是，apt只能识别Filer，不能识别一般的PrintWriter。
  • 创建一个工厂类(实现AnnotationProcessorFactory接口)，为注解指明正确的处理器，该接口中有三个方法：
    • getProcessorFor()方法用于返回注解处理器实例
    • supportedAnnotationTypes()用来检查是否所有的注解都有相应的处理器
    • supportedOptions()是否所有的控制台输入的参数(使用-A来指定)都是你提供支持的附加选项
  • 在命令行执行apt -factory 工厂类的完整路径 要进行处理的类的.java文件 -s 产生文件的输出目录(命令的运行格式是apt -classpath bin -factory annotation.apt.AssignmentApf src/annotation/work/*.java，即通过-factory来指定注解处理器工厂类的名称。实际上，apt工具在完成处理之后，会自动调用javac来编译处理完成后的源代码。)

apt的工作过程

从整个过程来讲，首先APT检测在源代码文件中哪些annotation存在。然后APT将查找我们编写的AnnotationProcessorFactory接口的实现类，并且要求AnnotationProcessorFactory实现类提供处理源文件中所涉及的annotation的annotation processor。接下来，一个合适的annotation processors将被执行，如果在processors生成源代码文件时，该文件中含有annotation，则APT将重复上面的过程直到没有新文件生成。

使用AbstractProcessor处理编译时注解(jdk1.7之后)

MyProcessor

这里有四个非常重要的方法：

  • init(ProcessingEnvironment env)：这个方法是用来初始化的注解处理器的，他会被注解处理工具调用，并且输入ProcessingEnvironment参数。ProcessingEnvironment提供了很多有用的工具类Elements，Types和Filer等，这些工具会在process方法中用到。
  • process(Set<? extends TypeElement> annoations, RoundEnvironment env):相当于每个注解处理器的主函数main()。在这些可以写一些扫描、判断，处理注解以及生成java文件的代码。输入参数RoundEnvironment，可以让你查询出包含特定注解的被注解元素。
  • getSupportedAnnotationTypes():该方法用于指定注解处理器可以处理的所有注解的类型，它的返回值是一个字符串集合。
  • getSupportedSourceVersion():用来指定使用的Java版本。通常返回SourceVersion.latestSupported()。

注解处理器是运行在自己的Java虚拟机中，javac会启动一个完整的java虚拟机来运行注解处理器。

注册注解处理器

将你自己编写的注解处理器MyProcessor使用jar命令打包成一个.jar文件，除此之外，需要将javax.annotation.processing.Processor文件(可以通过给注解处理器添加@AutoService(Processor.class)注解来自动生成，里面包含了注解处理器的合法的全名列表)到META-INF/services路径下，所以jar文件的文件结构如下所示：

  • MyProcessor.jar
    • 包结构…
      • MyProcessor.class
    • META-INF
      • services
        • javax.annotation.processing.Processor
          将你的MyProcessor.jar放到你的buildpath中，这样javac会自动检查和读取javax.annotation.processing.Processor中的内容，并且注册MyProcessor为注解处理器。

Elements,Types,Filer,Messager

  • Elements：一个用来处理Elements(Element代表程序的元素，例如包、类或者方法，每个Element代表一个静态的、语言级别的构件)的工具类
  • Types：一个用来处理TypeMirror(内部包含类的具体信息)的工具类，可以通过elements.asType()来获取对应元素的TypeMirror。
  • Filer：可以使用它来创建文件。
  • Messager：提供给注解处理器一个报告错误、警告或者提示信息的途径。Messager可以指定消息的不同级别，最严重的级别是Kind.ERROR,这种类型信息表示我们的注解处理器处理失败了。在出现错误时，注解处理器最好不要直接抛出一个异常，因为那样将会导致程序崩溃掉，保证注解处理器完成运行运行而不崩溃是更重要的。

处理循环

注解的处理过程是一个有序的循环，在每次循环中，一个处理器可能被要求去处理那些在上一次循环中产生的源文件和类文件中的注解(这里要注意你不能重载或者重新创建已经生成的源代码，所以在每次循环生成完文件之后，要清除掉已经写入到文件中的数据，是清除数据而不是清除文件)。第一次循环的输入是运行此工具的初始输入。这些初始输入，可以看成虚拟的第0次循环的输出。

process()方法一般的执行过程

  1. 搜索所有被指定注解A，得到使用了注解A的元素集合Set
  2. 检查被指定注解A注解的元素是类(TypeElement),方法,接口还是变量，在判断时要尽量避免使用instanceof，因为对此来说接口和类是相同的，要通过配合TypeMirror使用ElementKind或者TypeKind来处理。
  3. 使用Messager来进行错误信息的报告。
  4. 判断相应的匹配条件
  5. 使用Filer生成新的.java文件

异常和错误

  • AnnotationTypeMismatchException：当注解经过编译(或序列化)后，注解类型改变的情况下，程序视图访问该注解所对应的元素，则抛出此异常。
  • IncompleteAnnotationException：当注解经过编译(或序列化)后，将其添加到注解类型定义的情况下，程序视图访问该注解所对应的元素，则抛出此异常。
  • AnnotationFormatError：当注解解析器试图从类文件中读取注解并确定注解出现异常时，抛出该错误。

注解在Android中的应用:ButterKnife

ButterKnife的工作流程

在编译你的Android工程时，ButterKnife工程中ButterKnifeProcessor类的process()方法会执行以下的操作：

  • 1.开始它会扫描Java代码中的所有ButterKnife注解，然后生成以TypeElement为Key，BingingClass为value的Map;
  • 2.遍历生成的Map，当发现其中含有任何一个ButterKnife支持的注解后，会通过Filer来生成对应辅助类的.java文件，这个辅助类是以$$ViewBinder为后缀的，并且实现了ViewBinder接口(bind(包含了控件的实例化和设置点击事件等操作的代码)和unbind方法)
  • 3.最后当Activity启动，ButterKnife.bind(this)执行时，会根据类对应的Class类型去查找对应类的ViewBinder类是否存在于缓存(是使用LinkedHashMap来实现缓存的)中，如果不存在就通过反射创建该ViewBinder类的实例，并将其加入到LinkedHashMap中进行缓存，如果存在就直接从缓存中取出来，然后调用它的bind方法。
  • 4.在bind方法中，会进行控件实例化和设置点击事件等操作。

就这样我们就可以通过bind()方法+注解的方式来简化我们的代码了。
